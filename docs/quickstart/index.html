Quick Start

Overview

Onshape was built from the ground up as a true multi-tenant Software as a Service (SaaS) based system, and it uses REST APIs to communicate with its clients and third-party systems. In short, you call a REST API as you would do on a web page. It has a specific web address and often some additional parameters that are required either in the URL itself or a form of data that can be attached to the web call. The web call will return information instead of returning a web page, and the response from the API call is formatted in JSON (JavaScript Object Notation).

The REST API

The REST API stands for the REpresentational State Transfer Application Programming Interface. In Onshape, you can use the REST API to access Onshape engine and data in real-time. There are currently three types of REST API requests that are supported in Onshape (following the HTTP standards):

  • GET: Retrieve (read) information from the server (i.e., your Onshape documents). Any arguments included in the URL are sent to the server.
  • POST: Update (write) the server with new information. Data required is included in the body of the request â€“ the same as how a web form submits data to a server.
  • DELETE: Delete information from the server.

The Onshape Architecture

When you start a design in Onshape, you typically create a document. The document is the container that includes all the content related to a specific design. All data in an Onshape document is stored in Elements (represented as tabs in the user interface). Typically, Part Studios and Assemblies are two of the most common element types that a design requires. Throughout the design process, you may also find creating versions to be useful for product development management while working on the “Main” workspace. More details can be found in the Associativity page of the documentation.

Since Onshape is a web-based solution, it uses a URL to define what gets loaded in the browser. We can access different documents and different parts/versions of a document by simply altering the URL following a similar structure.

As an example, here is a typical Onshape URL:

https://cad.onshape.com/documents/58d051743d98a90f86e9e2ab/w/17e0ba65df28e4e72f7f4961/e/4a289bd094a999329a3900af
  • https://cad.onshape.com is the base URL of the document. It defines the Onshape Enterprise that the document belongs to. While the default for standard accounts is cad.onshape.com, every Enterprise account has a different base URL (e.g., companyName.onshape.com).
  • documents/ or simply d/ is followed by a 24-character document ID. The document ID uniquely identifies an Onshape document.
  • w/ is followed by a 24-character workspace ID. The workspace ID uniquely identifies a workspace within the document. By default, a document is started with the Main workspace.
    • But you can also create versions along the design process, which can be linked to by replacing w/ with v/, followed by a 24-character version ID.
    • With every edit made in the document, a microversion is automatically created internally. You can also access a specific microversion by replacing w/ with m/, followed by a 24-character microversion ID.
    • Note: POST requests will always be made to a workspace, as versions and microversions are immutable.
  • e/ is followed by a 24-character element ID. The element ID uniquely identifies an element (i.e., a tab that we see and access through the user interface).

Glassworks Onshape API Explorer

Onshape documents all available REST API endpoints on the Glassworks API Explorer, which can be reached through https://cad.onshape.com/glassworks/explorer/. It also provides the ability to run the API requests within its interface with the “Try it out” button and provides the output from the API call.

For a quick start, you can try out the API calls on Glassworks by running Onshape in a browser tab. Glassworks can use the logged-in Onshape session Cookie to authenticate the API calls. However, if you are making API calls to an Enterprise account, change the base URL of the Glassworks API explorer to the base URL of your Enterprise (e.g., https://companyName.onshape.com/glassworks/explorer/).

Making an Onshape REST API call

While making API calls on Glassworks is a great platform for testing out individual endpoints, there is a much greater potential on what the REST API can achieve by establishing computation workflows and developing web applications with APIs.

Authenticate access

To use the Onshape REST APIs outside of Glassworks, there are two possible workflows to authenticate the access to your Onshape account:

  • API Keys: allow a user (specifically, a developer) to make API requests
  • OAuth: allows an application to make API requests on behalf of the user

For a quick starting example, you can go through the following steps to create a set of API keys. More information about setting up the OAuth for app development can be found in the OAuth page.

  1. Go to the Onshape developer portal at https://dev-portal.onshape.com and log in with your Onshape account.
  2. Under “API keys” of the left panel, click “Create new API key” in the top right corner of the page.
  3. Choose the company and permissions that you would like this key to have access to (recommend enabling read-only permission for now, then be careful with the rest before you gain sufficient experience).
  4. Once the API key is created, a pop-up box should show an access key and a secret key. Note that the secret key will only show up once, or you will have to generate a new set of API keys if you lose access.

CAUTION: you should NOT share your API keys with others, nor upload and/or share the keys publicly. Gaining access to your API keys is equivalent to gaining access to your Onshape account with the permissions you specified in step 3 above. It is also good practice to periodically delete your API keys and generate a new set, such that you can deactivate any API keys that were leaked accidentally.

Introduction to a typical API call

For a typical REST API call in Onshape, there are five major components:

  • method: This should be one of GET, POST, and DELETE. Every API endpoint on Glassworks is labelled with its corresponding method type.
  • url: The URL specifies the API endpoint that is being called and the part of the document that the API is calling. The url should be in the following format: {base_url}/api/{fixed_url}. The {base_url} should be the same base URL that is introduced above (e.g., https://cad.onshape.com for non-Enterprise accounts). Then, the {fixed_url} is the URL that the API endpoint is titled with on Glassworks. However, you should replace all the information in {...} with the actual IDs.
  • query parameters: Besides the required parameters for the url, there are sometimes a few more optional paramters for the API call. In the drop-down content of every API endpoint on Glassworks, you can find a list of all optional query parameters with description.
  • headers: The header of an API request defines the associated meta-data, which is used by the server to process the request. Usually, this contains the "Content-Type" and "Accept". While the content type is generally set to be "application/json", the accept type can be found under “Media type” of the response section of every API endpoint on Glassworks.
  • payload body: A request body is only applicable for POST requests, and it’s a required component for some POST requests and optional for others. Again, a template of the payload body can be found under the drop-down content of the API endpoint on Glassworks. In most cases, the payload body should not be manually filled, but it should be obtained and modified from the response of a related GET request.

Example of making an API call

A REST API call for Onshape can be made the same way as you make an API call for most other platforms/servers. Samples codes in a few different popular programming languages can be found on the Sample Code page.
