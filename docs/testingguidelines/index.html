Testing Guidelines

The purpose of this document is to help you get your application and App Store entry ready for QA testing.

Application Release Workflow (ARW)

Each application submitted to the Onshape App Store goes through a series of stage-gates.

  • Ok to deploy to Production (Beta testing) (limited visibility)

  • Ok to make Public

  • Application is Public

The starting state is ‘Ok to deploy to limited visibility on Production’’. The end goal is ‘Application is Public on Production’. To advance to the next stage, your application must pass testing and your app store entry must pass review.

Testing may be requested at any time.

Testing is done on a first-come basis.

When testing is complete (pass or fail), you go to the back of the queue.

Testing protocol

Testing of Partner Applications will be performed against the checklist in addendum A.

Testing of Production App Store entries will be performed against the checklist in addendum B.

The result of each test will be one of;

  • Pass - no action needed and no notification will be issued

  • An enhancement - these are suggestions that should you chose to implement them, we believe would make the application better.

  • A bug (low priority) - these are slight deviations from the criteria but have low end user impact and will not prevent the application being turned on for public access and there is no stipulated time-frame for their resolution

  • A bug (medium priority) - these are material deviations from the criteria that are noticeable to the end-user and represent a minor problem requiring a work-around. A medium priority bug will not prevent the application from being turned on for public access, but should be fixed within 30 days.

  • A bug (high priority) - this is a significant deviation from the criteria and will prevent the application from being turned on for public access but will not prevent the application from being promoted from the partner stack.

  • A bug (MUST FIX) - this category usually represents a significant deviation from protocol or security violation. A MUST FIX will prevent an application on Production being turned on for public access and MAY, if the application is already public, cause it to be temporarily suspended from the app store.

Any test may be annotated with a ‘*’.

This indicates that the test is a must-pass in order for the application or app store entry to be promoted. A failure of this test will generate a bug (MUST FIX).

Actions that you can initiate from JIRA

  • Request application testing - This puts your application in the testing queue. We will note when testing has started (‘in progress’) and when concluded, the ticket will be closed. The outcome will include notes and links to any issues generated (‘tickets’). This phase may include as many iterations as needed to get your application ready. If necessary, the Application Release Workflow (ARW) will be updated.

  • Request public (general) authorization. We will note when testing has started (‘in progress’) and this request will trigger a review of your app store entry and any outstanding bugs. Note there is no implied testing of your application, simply a review of outstanding issues (tickets) and of the app store entry. Success at this stage will advance the Application Release Workflow (ARW) and you will now be able to request public (general) release.

  • Request public (general) release. This request states that you have coordinated with the Partner Application Manager (Joe Dunne) and the Onshape marketing department and general agreement has been reached that now is the time - Congratulations!

Addendum A - Application Testing Criteria

  1. Applied to all Applications
  2. Applies to Integrated Cloud Applications
  3. Applies to Connected Cloud Applications
  4. Applies to Connected Desktop Applications

x Denotes that this is a ‘must-pass’. A failure of this test results in a must-fix

  • (1)x The application must use the Onshape OAUTH mechanism
  • (1)x The OAUTH must be against the correct stack
  • (2)(3)x To be promoted to the Production stack, and hosted service must be on a monitored production server with worldwide 24/7 availability.
  • (2)(3) The application should not generate any avoidable console (browser) errors
  • (1) The application should should provide one or more of the following options;
    • Login using the Onshape ID (account created silently on first use)
    • Login with partner product account credentials
    • Create a new partner account
    • The user should not have to leave the registration workflow to complete a pre-requisite
  • (1) The application must be capable of managing/displaying documents in excess of 20 - suggested strategies include;
    • Use of a ‘Next’ button to load the next 20
    • Infinite scroll (loading the next 20 if/when the scrollbar reaches the bottom of the dialog
    • The application must display ‘reasonable’ performance when reading documents, workspaces, elements and parts. Consider at-scale than an account may have thousands of documents, many with multiple workspaces and each with multiple elements. Strategies may include any combination of;
    • Displaying the most-recently-opened first
    • Displaying a counter of documents/workspaces/elements read
    • Progressive loading
  • (1) The application should correctly list valid documents when ‘per document app access’ is turned on.
  • (1) The application should correctly handle selection of versions
  • (1) The application should correctly handle selection of workspaces (branches)
  • (1) The application should correctly handle/display elements that;
    • Are part studios that contain nothing
    • Are assemblies that contain nothing
    • Are part studios that contain only surfaces
    • Are part studios that contain only ‘wire’ data (eg helices)
  • (1) The application should appropriately handle revocation of a grant

Addendum B - App Store Testing Criteria

  1. Applied to all Applications
  2. Applies to Integrated Cloud Applications
  3. Applies to Connected Cloud Applications
  4. Applies to Connected Desktop Applications
  5. Required for deployment to production
  6. Only where the application is using Onshape billing

x Denotes that this is a ‘must-pass’. A failure of this test results in a must-fix

  • (1) The application should have a descriptive name
  • (1) The application summary should be accurate
  • (1)x The redirect URLS should be valid
  • (2)x The iframe URL should be valid
  • (1) The Grant (permissions) request should be no more than is needed
  • (1) The Application Type should be correctly set
  • (1) Team Visibility should be set
  • (1) The Category should be appropriate
  • (1) The application description should be accurate
  • (1) The Sign-In URL should be valid
  • (1) The pricing summary should be accurate. For instance, it cannot say ‘Free’ if this is a trial - it should say ‘Free for 30 days and then $30/month
  • (1) Any pay-plans should have an accurate description
  • (1) The support URL should point to a resource for help - this can be;
    • A support ticketing system (eg Zendesk)
    • A page giving a telephone number
    • A page giving an email address
    • A forum
    • It cannot be to an FAQ page (unless that page also contains one of the other options)
  • (1)The EULA link should point to an English Language EULA
