Model Translation

Summary

This document describes the APIs provided by Onshape for model translation. This document makes reference to specific APIs using the naming convention Group Name - Api Name, where the identifies an API group and identifies a specific API defined in the group. See the Glassworks API Explorer for details on the APIs.

Onshape provides several APIs to support model format translation. These fall into 3 basic categories

  1. Simple conversions - these support translation of Onshape content into STL and Parasolid format.
  2. General translation - these support translation from a variety of formats into a variety of other formats.
  3. Import translation - these translation operations are performed by default when a translatable file is uploaded to a blob element

Simple conversions

Simple conversions are limited to generating Parasolid and STL output. Most of the interfaces defined here operate by requesting an HTTP redirect to a different URL where the request will be fulfilled. Applications must be sure that they are explicitly handling the redirect and attaching authentication headers to the follow-up request or it will fail.

  • STL-specific

    Onshape Part Studio tabs and individual parts can be translated to STL format using the dedicated APIs

    • Part Studios - exportPartStudioStl
    • Parts - exportStl

    These interfaces require a Part Studio element Id and all use redirect behavior.

  • Parasolid-specific

    Onshape Part Studio tabs and individual parts can be translated to Parasolid format using the dedicated APIs

    • Part Studios - exportParasolid
    • Parts - exportPS

    These interfaces also require a Part Studio element Id and all use redirect behavior.

  • Multi-purpose

    In addition to the format-specific APIs defined for Parts and Part Studios, there are two similar multi-purpose APIs that can convert multiple source types to both STL and Parasolid.

    • Documents - export2Json (POST)

      This form requires the caller to perform an HTTP POST operation followed by an HTTP GET to a URL returned by the POST operation. There is no HTTP redirect performed; the application must explicitly follow the returned link.

    These APIs allow conversion of Assemblies as well as Parts and Part Studios into Either STL or Parasolid. The primary reason for the existence of the POST form of the API in addition to the GET form of the API is to overcome limits in URL and HTTP header length that can be exceeded when using long lists of part Id values.

General translations

The simple conversions listed above perform the format conversion synchronously, returning the output immediately after some processing delay. Other format conversions are more complex and time-consuming, and in many cases, cannot be completed quickly enough to prevent connection errors. The solution we use instead is a model where the application performs a 3-step process:

  1. HTTP POST operation to create a translation request
  2. Poll for operation completion or wait for webhook notification of completion
  3. HTTP GET operation to retrieve the translation result.

Translation Formats

The general translation APIs do not support STL, but do support a wide range of industry formats.

Output Formats

Supported output formats currently include PARASOLID, STEP, ACIS, IGES, SOLIDWORKS, ONSHAPE.

Source Formats

We allow a larger range of input formats that output formats. In addition to the formats listed above for input format, you can also convert from Part Studio, Assembly, PROE, NX, CATIA and INVENTOR.

APIs

We provide APIS to allow programatically determining supported formats:

  • Translations - getAllTranslatorFormats

    This API presents the full list of supported translation formats. For a more specific list of formats by version or workspace, see getElementTranslatorFormatsByVersionOrWorkspace.

Note that the source format for a translation is currently always automatically detected by Onshape. Part Studios and Assemblies are known to be ONSHAPE format. File uploads have their type determined by the filename suffix. For example, a file named part7.step is assumed to be in STEP format.

Translation initiation

Translation is initiated by performing an HTTP POST operation to one of the “Create Translation” APIs.

These APIs have some common options:

  • formatName

    This is the name of the format to which the source will be translated. This formatName must be one of the supported names (names are case-sensitive). Note that translation to ONSHAPE format requires storeInDocument=true.

  • storeInDocument

    Allows the caller to decide whether the output should be part of the document (as a Part Studio, Assembly or Blob tab) or whether the output should be external to the document.

APIs which use an upload or blob element common options when translating to ONSHAPE:

  • flattenAssemblies

    If the source file contains an assembly and flattenAssemblies is set to true, the assembly structure is removed and a single part studio is created.

  • yAxisIsUp

    Allows the application to tell Onshape translation whether the Y axis or Z Axis is the “Up” direction.

APIs

  • Translations - createTranslation

    This API allows an application to provide an input source file that will be converted to another format without requiring that the uploaded file be stored as a blob tab in the document. The API requires a document to be provided to define ownership context, but the uploaded file is not stored in the document.

    The file is uploaded as multipart form data. Each of the API parameters is a separate field in the multipart upload. The two required fields are “file” and “formatName”. The file part must contain the upload and should normally have a filename attribute set. The filename attribute can be omitted if the form data also has an encodedFilename field that contains the filename in URL-encoded form.

  • Blob Elements - createBlobTranslation

    A blob element containing a translatable file can be translated using this API. The blob element can be the result of a previous upload or it can be the output of a previous translation operation.

    • POST Body Example:

    The example below uses the STEP format. To view other acceptable formats, see getAllTranslatorFormats

    {
  "flattenAssemblies": false,
  "formatName": "STEP",
  "storeInDocument": true
}

  • Part Studios - createPartStudioTranslation

    Initiate translation for a Part Studio element

    • POST Body Example:

    The example below uses the OBJ format. To view other acceptable formats, see getAllTranslatorFormats

    {
  "flattenAssemblies": false,
  "formatName": "OBJ",
  "storeInDocument": false
}

  • Assemblies - translateFormat

    Initiate translation for an Assembly element

    • POST Body Example:

    The example below uses the STEP format. To view other acceptable formats, see getAllTranslatorFormats

    {
  "flattenAssemblies": false,
  "formatName": "STEP",
  "storeInDocument": true
}

The response from these APIs includes the fields:

  • id

    An identifier for the created translation operation. It can be used for later retrieval of translation results

  • href

    A URL for the Translations - Get Translation Status API for this specific translation

  • requestState Identifies the state of the translation request. The request will be done when the state changes from ACTIVE to DONE or FAILED. If FAILED, the translation was not successfully completed. If DONE, it completed successfully.

Waiting for translation completion

The simplest way to determine when the translation is complete is to poll. You do this in one of two ways (which are really the same thing):

  • Call the href returned in the translation response, which returns an updated state object
  • Call the Translation - Get Translation Status API, using the id returned in the translation response

Once the requestState changes to some value other than ACTIVE you will be ready to retrieve results.

An alternate way to wait for translation completion is to register a webhook and wait for a notification to be delivered. This is discussed later.

Retrieving results

Once a translation requestState becomes DONE, results are available to be used. How they are accessed depends on the specifics of the translation. A completed request has several additional fields that may be populated. If your original request specified storeInDocument=false then the results are stored as external data, in which case the resultExternalDataIds status field provides identifiers to be used for retrieval (see External Data, below). If the request specified storeInDocument=true then the resultElementIds field will be populated (see Element Data, below).

External Data

External data is data that is associated with, but external to the document used as translation context. This data is not versioned as is done with in-document data. A translation request to a format other than ONSHAPE currently always produces a single data file (from resultExternalDataIds).

To retrieve an external data file, use the API Documents - Download External Data

Element Data

When storeInDocument=true is specified, the results are stored as new elements in the document. The new elements are reported in the resultElementIds field in the translation status response. If the selected format is ONSHAPE, these elements will be Assemblies/Part Studios, and there may be multiple created elements. If the selected format is something other than ONSHAPE, there will be a single blob element result.

A blob element result can be extracted using the Blob Elements - Download File From Blob Element API.

Import Translation

When uploading a file to a blob element, either as a new element or an update to an existing element, if the file is a recognized format for import, it will, by default, be converted into ONSHAPE format. This behavior can be overridden by the application, if desired.

Webhook Translation Completion notification

If an application runs in a server with a network address that is reachable across the internet, the application may choose to use webhooks to find out about translation completion. For reliable operation, a webhook should be registered for the translation document prior to initiating translation. If registered after translation initiation, the notification might be lost. See the help document on Webhook Notifications for more details.
